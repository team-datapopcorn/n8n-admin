# 운영 관리자

n8n 워크플로우의 모니터링, 디버깅, 실행 관리를 담당하는 모듈.

## 모니터링

### 전체 현황 파악
1. `n8n_list_workflows`로 전체 워크플로우 목록 조회
2. 각 워크플로우의 활성/비활성 상태 확인
3. 워크플로우 수, 활성 비율 요약

### 실행 이력 분석
1. `n8n_executions`로 최근 실행 이력 조회
2. 성공/실패/대기 상태별 분류
3. 자주 실패하는 워크플로우 식별
4. 실행 시간 패턴 분석 (느린 워크플로우 감지)

### 상태 보고 형식
```
## n8n 워크플로우 현황

### 전체 요약
- 전체 워크플로우: N개 (활성: X개, 비활성: Y개)
- 최근 24시간 실행: N회 (성공: X회, 실패: Y회)

### 주의 필요 워크플로우
| 워크플로우 | 상태 | 최근 실패 | 실패율 |
|---|---|---|---|
| ... | ... | ... | ... |
```

## 디버깅

### 실패 원인 분석 프로세스
1. `n8n_executions`로 실패한 실행 조회
2. 에러 메시지와 실패한 노드 식별
3. `n8n_get_workflow`로 해당 워크플로우 구조 확인
4. 원인 분류:
   - **연결 에러**: 외부 서비스 접근 불가 (API 키 만료, URL 변경 등)
   - **데이터 에러**: 예상과 다른 입력 데이터 형식
   - **로직 에러**: 조건 분기 오류, 누락된 필드 처리
   - **리소스 에러**: 타임아웃, 메모리 초과

### 자동 수정 시도
1. `n8n_autofix_workflow`로 자동 수정 가능 여부 확인
2. 자동 수정이 적용되면 결과 검토
3. 자동 수정 실패 시 수동 수정안 제시

### 수동 수정 적용
1. 수정안을 사용자에게 구체적으로 설명
2. 사용자 승인 후 `n8n_update_partial_workflow`로 패치
3. 수정 후 `n8n_validate_workflow`로 검증
4. 가능한 경우 `n8n_test_workflow`로 테스트

### 롤백
- 수정이 실패하거나 상황이 악화되면 `n8n_workflow_versions`의 rollback 모드 사용
- 롤백 전 사용자에게 "이전 버전으로 되돌리겠습니다" 확인

## 실행 관리

### 수동 테스트
- `n8n_test_workflow`로 즉시 실행 (webhook/form/chat 트리거만)
- 테스트 데이터가 필요한 경우 사용자에게 요청하거나 샘플 데이터 제안
- 스케줄 트리거 워크플로우는 "n8n UI에서 'Test workflow' 버튼을 눌러주세요" 안내

### 워크플로우 삭제
- `n8n_delete_workflow` 실행 전 반드시 확인:
  - "워크플로우 '[이름]' (ID: [ID])을 삭제하시겠습니까? 이 작업은 되돌릴 수 없습니다."
- 삭제 전 `n8n_workflow_versions`로 백업 가능 여부 안내

### 버전 관리
- `n8n_workflow_versions`로 변경 이력 조회
- 특정 버전으로 rollback 가능
- 중요 변경 전 현재 버전 번호 기록

## 핵심 원칙

1. **안전 우선**: 파괴적 작업은 항상 사용자 확인 후 실행
2. **증거 기반**: 실행 로그와 에러 메시지를 기반으로 판단, 추측 최소화
3. **자동 → 수동 에스컬레이션**: autofix 먼저 시도, 실패 시 수동 수정 제안
4. **롤백 대비**: 수정 전 현재 버전 확인, 실패 시 즉시 복원 가능하도록 준비

## 에러 트리거 관리

### 현황 조회
"에러 트리거 현황 보여줘" 요청 시:
1. `n8n_list_workflows`로 전체 워크플로우 조회
2. 각 워크플로우의 nodes에서 `n8n-nodes-base.errorTrigger` 존재 여부 확인
3. 결과를 표로 출력:

| # | 워크플로우 | 상태 | 에러 트리거 |
|---|----------|------|-----------|
| 1 | [CRON] 매일 아침 AI 브리핑 | 활성 | 있음 |
| 2 | [WEBHOOK] 고객 문의 분류 | 활성 | **없음** |

### 일괄 등록
"전체 워크플로우에 에러 트리거 등록해줘" 요청 시:
1. 에러 트리거가 없는 활성 워크플로우 목록을 보여주고 확인을 구한다
2. 확인 후 각 워크플로우에 Error Trigger 노드를 추가한다:
   - `n8n_get_workflow`로 현재 상태 조회
   - nodes 배열에 errorTrigger 노드 추가
   - `n8n_update_full_workflow`로 저장
3. 결과 리포트 출력

## 퇴사자 인계

### 소유 자산 조회
"홍길동 퇴사 인계 처리해줘" 또는 "user@email.com의 자산 보여줘" 요청 시:
1. `n8n_list_workflows`로 전체 워크플로우 조회
2. shared 필드에서 해당 유저의 personal project가 owner인 워크플로우 필터링
3. 결과를 표로 출력:

| # | 유형 | 이름 | 상태 |
|---|------|------|------|
| 1 | 워크플로우 | [CRON] 매일 아침 AI 브리핑 | 활성 |
| 2 | 워크플로우 | [WEBHOOK] 고객 문의 분류 | 활성 |
| 3 | 크레덴셜 | OpenAI API | - |

### 소유권 이전
인수자를 확인한 후:
1. 이전할 자산 목록과 인수자를 보여주고 확인을 구한다
2. 워크플로우: n8n API `PUT /api/v1/workflows/{id}/transfer` (body: `{ destinationProjectId }`)
3. 크레덴셜: n8n API `PUT /api/v1/credentials/{id}/transfer` (body: `{ destinationProjectId }`)
4. 결과 리포트 출력

### 휴면 유저 위험 감지
"휴면 유저 중 워크플로우 가진 사람 보여줘" 요청 시:
1. 유저 목록에서 90일+ 미접속 유저 필터링
2. 각 휴면 유저의 소유 워크플로우 개수 조회
3. 워크플로우를 보유한 휴면 유저만 표로 출력

## 결과 출력 포맷

모니터링/분석 결과는 항상 표 형식으로 정리하여 보여준다.

**워크플로우 현황 예시:**

| # | 이름 | 상태 | 최근 실행 | 결과 |
|---|------|------|----------|------|
| 1 | [CRON] 매일 아침 AI 브리핑 | 활성 | 2026-03-26 09:00 | 성공 |
| 2 | [WEBHOOK] 고객 문의 분류 | 활성 | 2026-03-26 14:32 | 실패 |

**에러 리포트 예시:**

| # | 워크플로우 | 에러 타입 | 발생 시각 | 원인 요약 |
|---|----------|----------|----------|----------|
| 1 | 고객 문의 분류 | NodeOperationError | 03-26 14:32 | API 키 만료 |

**유저 현황 예시:**

| # | 이메일 | 역할 | 최근 접속 | 미접속 일수 | 상태 |
|---|--------|------|----------|-----------|------|
| 1 | admin@co.kr | Owner | 오늘 | 0 | 정상 |
| 2 | dormant@co.kr | Member | 2025-10-01 | 177 | 휴면 |
