# 워크플로우 빌더

자연어 요청을 n8n 워크플로우로 변환하고, 기존 워크플로우를 분석/개선하는 모듈.

## 워크플로우 생성 프로세스

### 1단계: 요청 분석
- 사용자 요청에서 트리거 유형, 처리 로직, 출력 대상을 식별
- 필요한 노드 목록을 추출

### 2단계: 노드 검색 및 확인
- `search_nodes`로 필요한 노드 검색
- `get_node`로 각 노드의 파라미터, 입출력 스펙 확인
- `validate_node`로 노드 설정이 유효한지 검증

### 3단계: 워크플로우 JSON 구성
- 노드 배열, 연결(connections), 설정(settings) 구성
- 노드 포지션 배치: 좌→우로 300px 간격, 분기 시 상하로 200px 간격
- 트리거 노드는 항상 가장 왼쪽 (position: [250, 300])

### 4단계: 사전 검증
- `validate_workflow`로 JSON 구조 검증
- 에러 발생 시 수정 후 재검증
- `n8n_autofix_workflow` 활용 가능

### 사용자 미리보기 (생성 전 확인)

생성 실행 전에 사용자에게 다음 정보를 표로 보여주고 확인을 받는다:

| 항목 | 내용 |
|------|------|
| 워크플로우 이름 | 제안된 이름 |
| 트리거 | 트리거 노드 타입 + 설정 요약 |
| 노드 구성 | 전체 노드 목록 (순서대로) |
| 연결 | 노드 간 연결 흐름 (A → B → C) |
| 초기 상태 | 비활성 (사용자가 수동 활성화) |

"위 구성으로 워크플로우를 생성합니다. 진행할까요?"

### 5단계: 생성 및 테스트
- `n8n_create_workflow`로 워크플로우 생성
- webhook/form/chat 트리거인 경우 `n8n_test_workflow`로 테스트
- 스케줄 트리거인 경우 "n8n UI에서 수동 실행으로 테스트해주세요" 안내

### 6단계: 결과 보고
- 생성된 워크플로우 ID, 이름, 노드 구성 요약
- 테스트 결과 (성공/실패 및 출력 데이터)
- n8n UI에서 확인할 수 있는 안내

## 복잡한 패턴

### 에러 핸들링
- Error Trigger 노드를 추가하여 워크플로우 실패 시 알림
- 개별 노드에 onError 설정: "continueOnFail" 또는 "stopWorkflow"
- 에러 알림 대상: 이메일, 슬랙, 텔레그램 등 사용자 선호에 따라 설정

### 조건 분기
- IF 노드: 단순 조건 (true/false 분기)
- Switch 노드: 다중 조건 분기 (3개 이상 경로)
- 분기 후 각 경로의 노드를 상하로 배치하여 시각적 구분

### 반복 루프
- SplitInBatches 노드: 대량 데이터를 배치 단위로 처리
- 배치 크기는 기본 10, API 제한에 따라 조정
- 루프 완료 후 Merge 노드로 결과 취합

### 서브 워크플로우
- Execute Workflow 노드로 다른 워크플로우 호출
- 공통 로직을 서브 워크플로우로 분리하여 재사용
- 파라미터 전달: JSON 데이터로 입력값 전달

## 기존 워크플로우 분석/개선

### 분석 프로세스
1. `n8n_get_workflow`로 워크플로우 구조 조회
2. 다음 항목 점검:
   - 에러 핸들링 누락 여부
   - 불필요한 중복 노드
   - 비효율적 데이터 처리 (전체 로드 vs 필터링)
   - 하드코딩된 값 (변수로 대체 가능 여부)
3. 개선안을 구체적으로 제시 (어떤 노드를 어떻게 변경할지)

### 개선 적용
- 사용자 승인 후 `n8n_update_partial_workflow`로 부분 수정
- 구조 변경이 큰 경우 `n8n_update_full_workflow` 사용
- 수정 전 `n8n_workflow_versions`로 현재 버전 기록

## 데이터 테이블 연동

- `n8n_manage_datatable`로 테이블 조회/삽입/업데이트
- 활용 예시: 상태 추적, 설정값 저장, 룩업 테이블
- 워크플로우 내에서 데이터 테이블 노드와 연결하여 영구 상태 관리

## 자주 쓰는 노드 조합 가이드

### 웹훅 → 처리 → 응답
- Webhook → Code/Set → Respond to Webhook
- API 엔드포인트 구축에 적합

### 스케줄 → 수집 → 알림
- Schedule Trigger → HTTP Request → IF → Slack/Email
- 주기적 모니터링에 적합

### 트리거 → 분기 → 다중 출력
- Webhook/Schedule → Switch → 각 경로별 처리
- 조건에 따라 다른 서비스로 데이터 전달

### 데이터 동기화
- Schedule Trigger → HTTP Request(소스) → Code(변환) → HTTP Request(대상)
- 서비스 간 데이터 동기화에 적합
